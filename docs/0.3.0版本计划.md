## 一、评价系统

### 1.1 概述

交易完成后，买家可以对商品进行评分（1~5 星），系统汇总计算商品平均评分和卖家综合评分。评分是可选行为——买家可以选择不评分。评分覆盖两类订单：**自主取消**和**购买成功**（即卖家确认完成）。

### 1.2 数据模型

**新增 `review` 表：**

```sql
CREATE TABLE IF NOT EXISTS review (
    review_id   BIGINT   NOT NULL AUTO_INCREMENT COMMENT '主键',
    order_id    BIGINT   NOT NULL COMMENT '关联订单ID',
    item_id     BIGINT   NOT NULL COMMENT '关联商品ID',
    reviewer_id BIGINT   NOT NULL COMMENT '评分人（买家）ID',
    target_id   BIGINT   NOT NULL COMMENT '被评人（卖家）ID',
    rating      TINYINT  NOT NULL DEFAULT 0 COMMENT '评分 0未评 1~5已评',
    content     VARCHAR(500) DEFAULT NULL COMMENT '评价内容（可选）',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (review_id),
    UNIQUE KEY uk_order (order_id),
    KEY idx_item (item_id),
    KEY idx_target (target_id)
) ENGINE=InnoDB;
```

- `order_id` 唯一约束：一笔订单只能评价一次
- `rating` 取值 0~5，默认 0。0 表示买家选择不评分，1~5 为有效评分。写入记录即代表买家参与了评价流程
- `content` 为可选评语，评分和评语均为可选

### 1.3 后端实现

#### 实体与 DTO

| 文件 | 说明 |
|------|------|
| `entity/Review` | 对应 `review` 表 |
| `dto/ReviewDTO` | 提交评价请求体：`orderId`、`itemId`、`rating`(0~5)、`content`(可选)，含 `@NotNull @Min(0) @Max(5)` 校验 |
| `vo/ReviewVO` | 评价展示：`reviewId`、`orderId`、`reviewerName`、`rating`、`content`、`createTime` |
| `vo/ItemRatingVO` | 商品评分摘要：`itemId`、`averageRating`(平均值，保留一位小数)、`reviewCount` |
| `vo/UserRatingVO` | 用户综合评分：`userId`、`overallRating`(所有商品的评分均值，排除 0)、`ratedItemCount`(有评分的商品数) |

#### Mapper / Service / Controller

| 层 | 文件 | 关键方法 |
|----|------|----------|
| Mapper | `ReviewMapper` + `ReviewMapper.xml` | `insert`、`getByOrderId`、`listByItemId`、`avgRatingByItemId`、`avgRatingBySellerId` |
| Service | `ReviewService` + `ReviewServiceImpl` | `submit`(校验订单属于当前用户且状态为已取消或已完成)、`getByItem`、`getByUser` |
| Controller | `ReviewController` | `POST /api/reviews`(提交评价)、`GET /api/reviews/item/{itemId}`(商品评价列表)、`GET /api/reviews/users/{userId}`(用户收到的评价) |

#### 评价入口判断逻辑

```
对订单状态判断：
  - orders.status = 2 (已完成) → 允许评价
  - orders.status = 3 (已取消) → 允许评价
  - orders.status = 1 (待确认) → 不允许评价
  
评价前检查：
  - 当前用户 == order.buyerId（只有买家能评价卖家）
  - review 表无同 order_id 记录（一笔订单只评一次）
  - rating 在 0~5 之间（0 表示选择不评分）
  - rating ≥ 1 时 content 可为空；rating = 0 时不强制写 content

评价写入：
  - 无论 rating 是否为 0，都写入 review 表（记录买家已参与评价流程）
  - 前端弹窗可以让用户主动跳过（关闭即 rating=0，或提供一个"跳过"按钮）
```

#### 商品评分计算

- 商品详情接口 `ItemDetailVO` 新增字段：`averageRating`（此商品的有效评分均值）、`reviewCount`（已评分人数）
- 计算方式：`SELECT AVG(rating), COUNT(*) FROM review WHERE item_id = ? AND rating >= 1`
- 无有效评分时 `averageRating` 返回 0，前端按"暂无评分"展示

#### 用户综合评分计算

- 用户主页新增综合评分字段：对该用户售出的所有商品的评分取均值（仅计入 rating ≥ 1 的评价）
- 计算方式：先查该用户售出的所有商品 ID，再对这些商品的 `review` 评分求 AVG（`WHERE rating >= 1`）
- 无评分时返回 0，前端展示"暂无评分"

## 二、用户主页完善

### 2.1 现状

- 商品详情页卖家卡片中有"查看主页"按钮，路由为 `/user/:id`
- 但当前 router 中没有 `/user/:id` 路由，前端无对应页面
- 后端 `GET /api/users/{id}` 返回用户基本信息，`GET /api/users/{id}/items` 返回该用户发布的商品（仅 `status=1` 在售）

### 2.2 后端改动

#### 用户商品接口增强

- `GET /api/users/{id}/items` 增加请求参数 `status`（可选），不传时返回全部商品（在售 + 已售出）
- `ItemVO` 新增字段：`averageRating`（商品平均评分）、`reviewCount`

#### 新增用户统计接口

- `GET /api/users/{id}/profile`：返回用户公开主页信息
  - 基本信息：`userId`、`username`、`avatar`、`studentId`（脱敏仅显示院系级信息或隐去）
  - 统计信息：`totalSold`(已售出数量)、`totalItems`(在售数量)、`overallRating`(综合评分)、`ratedItemCount`
  - 可以考虑放在 `UserController` 中作为新方法，也可以新建 `UserProfileController`

#### ItemDetailVO / ItemVO 补充评分字段

- `ItemDetailVO` 新增：`averageRating`(BigDecimal)、`reviewCount`(Integer)
- `ItemVO` 新增：`averageRating`(BigDecimal)、`reviewCount`(Integer)
- `ItemMapper.xml` 中 `getDetailById` 和 `getByUserId` / `pageQuery` SQL 使用子查询或 LEFT JOIN 获取 `review` 表评分数据

### 2.3 前端实现

**新增页面 `UserHomeView.vue`**（路由：`/user/:id`）：

| 区块 | 内容 |
|------|------|
| 用户信息头部 | 头像（首字母）、用户名、综合评分（⭐ X.X）、已售/在售统计 |
| 商品列表 | 分 tab 或不分 tab 展示所有商品卡片，每张卡片显示商品评分 |
| 空状态 | 该用户暂无商品时展示提示 |

**改动 `ItemDetail.vue`**：
- 卖家卡片中新增商品平均评分展示
- "查看主页"按钮可跳转到 `/user/{sellerId}`

**新增路由**：
```js

```

- 商品卡片底部新增星级评分展示（只读）

## 五、接口限流

### 5.1 方案

**基于 IP + Redis 滑动窗口的自定义 `@RateLimiter` 注解 + AOP 实现。**

**计划**：
- [ ] 自定义 `@RateLimiter` 注解：`key`(限流标识)、`limit`(最大请求数)、`windowSeconds`(时间窗口)
- [ ] `RateLimiterAspect` 切面：使用 Redis `ZSET` 实现滑动窗口计数
- [ ] 超出限制时抛出 `RateLimitExceededException`，`GlobalExceptionHandler` 捕获并返回 HTTP 429
- [ ] 在以下接口应用：
  - `POST /api/users/login`：10 次/分钟
  - `POST /api/users/register`：3 次/分钟
  - `POST /api/common/upload`：20 次/分钟
  - `POST /api/orders`：30 次/分钟

### 5.2 依赖

无需引入新依赖，基于 Spring AOP + Redis 即可。
